# 角色权限矩阵 V1 设计

日期：2026-04-21

适用范围：`retail_ops_system` 当前一期权限设计，仅覆盖以下角色：

- 仓库主管
- 区域运营经理
- 店长
- 店员
- 收银员

## 1. 设计目标

在单一系统内建立一套可执行的权限模型，确保以下四件事始终一致：

- 角色定义
- 设备入口
- 数据范围
- 页面与动作权限

本设计明确采用以下固定规则：

- One system, role/device-based visibility.
- Do not split into many separate products for each role.
- Manager and clerk interfaces must remain separate.
- Permissions should be defined by role + device + data scope + action.

## 2. 不在本次设计范围内

以下内容不在本次权限设计内：

- 具体业务页面实现细节
- 打印机 / PDA 联调
- 百货成本逻辑
- 老板经营分析大盘
- 中国录入、中国财务、HR、老板等未纳入本轮角色

## 3. 核心设计原则

### 3.1 单系统，多工作区

系统只有一个，不拆成多个产品。

但不同角色必须进入不同工作区：

- 仓库主管：仓库工作区
- 区域运营经理：区域管理工作区
- 店长：门店管理工作区
- 店员：门店执行工作区
- 收银员：POS 工作区

### 3.2 兼岗允许，但不混界面

同一个人可以兼任多个角色，例如：

- 店长 + 店员
- 店员 + 收银员

但兼岗只代表账户可切换多个角色，不代表界面混合。一次登录会话必须进入一个明确角色工作区，不允许把店长、店员、收银界面混在同一导航结构里。

### 3.3 权限四层结构

每个权限都必须同时经过以下四层约束：

1. 角色：你是谁
2. 设备：你从什么设备进入
3. 数据范围：你能看哪一部分数据
4. 动作：你能做什么事

对于 POS，再增加两层强校验：

5. 班次：你是否是该班收银员
6. 会话：该 POS 是否已被你开班锁定

## 4. 角色与设备策略

### 4.1 设备类型

当前权限设计只使用以下设备类型：

- 桌面端
- PDA / 手机
- POS 收银机

### 4.2 设备默认首页

| 角色 | 允许设备 | 默认首页 | 说明 |
| --- | --- | --- | --- |
| 仓库主管 | 桌面端 | 仓库控制台 | 仓库主作业与异常处理入口 |
| 区域运营经理 | 桌面端 | 区域运营控制台 | 区域调拨、巡店现金管理、新店筹划、跨店跟进与审批入口 |
| 店长 | 桌面端 | `5. 门店执行控制中心` | 门店管理与分配入口 |
| 店员 | PDA / 手机、桌面端 | PDA: `6.2 我的当前包 / 7. 店员工作台`；桌面: `7.1 打印/补打支持页` | 执行工作入口 |
| 收银员 | POS 收银机 | `9. 收银销售页` | POS 专用入口 |

## 5. 数据范围定义

| 角色 | 数据范围 |
| --- | --- |
| 仓库主管 | 所属仓库全部业务数据 |
| 区域运营经理 | 所辖区域全部门店数据，以及区域调拨单、门店现金交接/收现记录、新店筹划事项 |
| 店长 | 本店全部业务数据 |
| 店员 | 本店范围内，且仅限自己被分配任务、bale、商品、打印记录 |
| 收银员 | 本店范围内，且仅限当前班次、当前 POS、当前收银会话数据 |

补充规则：

- 店长不可看跨店明细。
- 区域运营经理可跨店看区域内门店，并管理区域调拨、现金巡收、新店筹划，但不可代替门店做店员实操、收银实操、仓库实操。
- 店员不可查看其他店员的工作明细。
- 收银员不可查看其他收银员班次明细，不可跨设备查看其他 POS 会话。

## 9. 收银台专项权限设计

### 9.1 设计原则

`收银员` 必须是独立角色，不并入 `店员`。

但收银权限不能只靠角色本身，而必须建立为：

- 独立角色
- 班次绑定
- 设备绑定
- 会话锁定

### 9.2 判定条件

用户只有在同时满足以下条件时，才允许收银：

1. 账户拥有 `收银员` 角色
2. 当天被排到该店、该 POS 设备、该班次
3. 已在该设备成功开班
4. 当前 POS 会话未被他人锁定

### 9.3 POS 强约束

- 当天不是该班收银员的人，不得进入销售操作态
- 同店其他收银员，即使也有 `收银员` 角色，只要不是该班，也不得操作该 POS
- 当日做客服、卫生等辅助工作的员工，即使在店里，也不得因此取得 POS 权限
- 店长默认不得顶替收银；若店长需要顶班，必须同时拥有 `收银员` 角色并被正式排班
- POS 设备登录后只进入 POS 工作区，不展示店员、店长菜单

## 10. 前端显示权限规则

前端权限只负责界面裁剪，不作为最终授权依据。

### 10.1 显示规则

- 登录后先按设备类型过滤可进入角色
- 若账户拥有多个角色，只显示当前设备允许的角色入口
- 进入角色后，只渲染该角色允许的菜单
- 不允许把店长页、店员页、收银页合并到同一个菜单结构
- 店员仅展示自己的任务、自己的 bale、自己的打印与上架入口
- 店长展示本店执行全局，但不默认进入店员工作台
- 收银员在 POS 只展示销售、退款、作废、支付异常、开班、交班相关入口
- 区域运营经理展示区域调拨、巡店现金管理、新店筹划、区域门店状态与异常跟进入口

### 10.2 建议入口结构

- 桌面端：
  - 仓库主管 -> 仓库控制台
  - 区域运营经理 -> 区域运营控制台
    - 内含：区域调拨、现金巡店、门店异常、新店筹划
  - 店长 -> 门店执行控制中心
  - 店员 -> 打印/支持页或我的当前包
- PDA / 手机：
  - 店员 -> 店员工作台
  - 若同一人兼任店长，也必须切到店长角色后才进入店长工作区
- POS 终端：
  - 只允许收银员进入 POS 工作区

## 11. 后端硬权限边界

后端必须对每个接口做硬校验，禁止仅依赖前端隐藏按钮。

### 11.1 统一校验顺序

1. 校验登录身份
2. 校验当前激活角色
3. 校验设备类型是否允许该角色
4. 校验该角色是否有页面/动作权限
5. 校验数据范围
6. 若是 POS，再校验班次、设备、开班会话

### 11.2 角色硬边界

#### 仓库主管

- 不可访问 POS 销售、退款、作废接口
- 不可修改门店售价、货架位、店员任务
- 不可查看与自己仓库无关的数据

#### 区域运营经理

- 可创建和审批区域调拨，但不可执行仓库入仓、分拣、打印、仓库出库等仓库动作
- 可管理区域巡店现金交接、收现记录与异常跟进，但不可直接执行 POS 销售
- 可创建和推进区域新店筹划事项，但不可替代门店做店员上架、打印、收银等实操
- 只能访问所属区域数据，不可越区管理

#### 店长

- 不可调用仓库分拣、发店放行、仓库打印接口
- 不可直接调用 POS 销售提交接口
- 不可查看跨店数据

#### 店员

- 只能修改自己被分配商品与任务
- 不可签收整店收货
- 不可分配 bale
- 不可审批
- 不可访问销售、退款、作废接口

#### 收银员

- 只能操作当前班次、当前 POS、当前店数据
- 不可访问上架、分配、签收、仓库操作接口
- 不可越权查看其他收银员班次和其他门店交易明细

## 12. 账户兼岗建议

V1 推荐采用“账户可兼岗、会话分角色”的方案：

- 一个账户可被赋多个角色
- 登录后根据设备类型展示可进入的角色工作区
- 用户进入某一角色工作区后，该会话只按该角色授权

推荐兼岗场景：

- 店长需要下场做上架：赋 `店长 + 店员`
- 店员需要轮流收银：赋 `店员 + 收银员`

不推荐做法：

- 仅因员工在店内，就自动开放收银权限
- 仅因员工是店长，就默认可进入 POS 操作页

## 13. V1 落地优先级

建议按以下顺序落地：

1. 固化角色代码、设备类型、数据范围枚举
2. 固化默认首页与菜单显示规则
3. 固化页面级权限与动作级权限矩阵
4. 为 POS 增加班次绑定、设备绑定、开班会话锁定
5. 把前端显示权限升级为后端硬权限校验

## 14. 本版结论

V1 权限结构的核心不是“谁能看到哪个页面”，而是建立一套一致的权限骨架：

- 用角色区分工作区
- 用设备区分入口
- 用数据范围区分可见对象
- 用动作权限区分可执行事项
- 用审批节点区分责任边界
- 用后端硬权限保证前端不可绕过

对当前五个角色而言，最终结构应保持为：

- 仓库主管 = 仓库工作区
- 区域运营经理 = 区域管理工作区
- 店长 = 门店管理工作区
- 店员 = 门店执行工作区
- 收银员 = POS 工作区

这是当前一期角色矩阵的固定口径。
