## Scope rules

- This matrix describes the existing warehouse sorting main flow.
- It does not approve any business-code, backend-route, frontend-prototype, React admin, runtime-data, cache, generated-output, or secret change.
- Direct-hang / direct-dispatch variants are exceptions, not the main warehouse sorting flow.
- Barcode business identity must remain strongly typed: `RAW_BALE`, `DISPATCH_BALE`, `STORE_ITEM`, `BALE_SALES`, and `LOOSE_PICK` are not interchangeable.
- `template_scope` is print-template classification only; it is not business identity.
- `identity_id` remains the final store-item lifecycle anchor.

## Status and barcode contract notes

| Contract area | Rule |
|---|---|
| Main-flow identity | The main warehouse flow starts with `RAW_BALE`, then creates sorted stock, then creates prepared dispatch/sale bales. It must not silently convert raw bale identity into store item identity. |
| Cost authority | `0.2 Sorting confirmation / cost lock` is the cost lock point. Backend must remain authoritative for cost lock and `cost_locked_at`. |
| Store authority | Store users may change sale price, display/rack location, and operational shelf placement only. They must not change warehouse cost or source lineage. |
| POS acceptance | POS may accept only `STORE_ITEM`. `RAW_BALE`, `DISPATCH_BALE`, `BALE_SALES`, and `LOOSE_PICK` must be rejected with operational next-step copy. |
| Bale sales | B2B whole-bale sale is separate from retail POS. Waiting-sale bales should move through bale-sales candidate/order flow, not store item checkout. |
| Return boundary | For store thread, return-to-warehouse is an endpoint. Further RET split / resale / scrap decisions remain a later warehouse policy decision, not part of this matrix. |
